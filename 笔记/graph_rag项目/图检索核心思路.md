**代码里的图 RAG 检索，不是先做向量相似度，而是先把自然语言问题翻译成“图查询意图”，再去 Neo4j 里找路径或子图，最后把路径/子图包装成 `Document` 给后续 LLM 使用。**

下面按真实执行链路一步一步讲。

**1. 初始化图检索器**

图 RAG 模块在 [graph_rag_retrieval.py](/d:/project/LearnAgent/all-in-rag/code/C9/rag_modules/graph_rag_retrieval.py) 里。

初始化时先连接 Neo4j：

```python
self.driver = GraphDatabase.driver(
    self.config.neo4j_uri,
    auth=(self.config.neo4j_user, self.config.neo4j_password)
)
```

连接成功后，会调用 `_build_graph_index()`。这个“索引”不是数据库底层索引，而是程序内存里的缓存：

```text
entity_cache: 记录高频实体节点
relation_cache: 记录关系类型及频次
subgraph_cache: 预留的子图缓存
```

它会从 Neo4j 中读取节点的 `nodeId`、`name`、`category`、度数，以及关系类型频次。作用是让系统知道图里大概有哪些实体、哪些关系类型。

这里的图结构大致是：

```text
Recipe 菜谱
Ingredient 食材
Category 分类
CookingStep 烹饪步骤

Recipe -[:REQUIRES]-> Ingredient
Recipe -[:BELONGS_TO_CATEGORY]-> Category
Recipe -[:CONTAINS_STEP]-> CookingStep
```

所以图 RAG 的检索对象不是“文本块”，而是这些节点和边。

**2. 用户问题进入 `graph_rag_search`**

主入口是 [graph_rag_search](/d:/project/LearnAgent/all-in-rag/code/C9/rag_modules/graph_rag_retrieval.py:482)。

它接收：

```python
graph_rag_search(query, top_k=5)
```

第一件事是检查 Neo4j 是否可用。如果没有连接，直接返回空列表。因为这个模块的核心证据都在图数据库里，没有 Neo4j 就无法做图遍历。

然后它不会立刻查库，而是先进入下一步：理解问题的图意图。

**3. 把自然语言问题转换成 `GraphQuery`**

这一步在 [understand_graph_query](/d:/project/LearnAgent/all-in-rag/code/C9/rag_modules/graph_rag_retrieval.py:141)。

它调用 LLM，让模型把用户问题解析成结构化对象：

```python
GraphQuery(
    query_type=...,
    source_entities=...,
    target_entities=...,
    relation_types=...,
    max_depth=...
)
```

这一步很关键，因为图数据库不能直接理解“鸡肉配什么蔬菜好”。系统需要先判断：

```text
这个问题是查直接关系？
还是多跳推理？
还是要拿一个完整子图？
还是找 A 到 B 的路径？
```

代码里定义了几种查询类型：

```text
ENTITY_RELATION: 实体关系查询
MULTI_HOP: 多跳查询
SUBGRAPH: 子图查询
PATH_FINDING: 路径查找
CLUSTERING: 相似/聚类查询
```

举个例子，用户问：

```text
鸡肉配什么蔬菜好？
```

LLM 应该解析成：

```json
{
  "query_type": "multi_hop",
  "source_entities": ["鸡肉"],
  "target_entities": ["蔬菜"],
  "relation_types": ["REQUIRES", "BELONGS_TO_CATEGORY"],
  "max_depth": 3
}
```

这表示：从“鸡肉”这个食材节点出发，沿着图里的关系走，找能连接到“蔬菜”相关节点的路径。

注意这里有个实现细节：prompt 里让 LLM 返回 `constraints`，比如低糖、30 分钟内，但代码构造 `GraphQuery` 时没有把 `constraints` 塞进去。所以当前实现对复杂属性过滤支持还不完整。

**4. 根据查询类型选择检索路线**

`graph_rag_search` 拿到 `GraphQuery` 后开始分流。

逻辑是：

```text
MULTI_HOP / PATH_FINDING
=> 走多跳路径检索

SUBGRAPH / CLUSTERING
=> 走子图提取 + 图结构推理

ENTITY_RELATION
=> 也走路径检索，只是理论上深度更浅
```

也就是说，图 RAG 不是一个单一检索算法，而是一组图查询策略。

**5. 多跳查询：从源实体出发找路径**

如果查询类型是 `MULTI_HOP`，进入 [multi_hop_traversal](/d:/project/LearnAgent/all-in-rag/code/C9/rag_modules/graph_rag_retrieval.py:263)。

它先取出：

```text
source_entities: 起点实体，比如 鸡肉
target_entities: 目标关键词，比如 蔬菜
max_depth: 最大跳数，比如 3
relation_types: 优先关系类型，比如 REQUIRES
```

然后构造 Cypher 查询。

核心思想是：

```cypher
MATCH path = (source)-[*1..max_depth]-(target)
```

这表示：从 `source` 出发，沿任意方向的关系，走 1 到 `max_depth` 跳，找到所有能到达的 `target`。

如果有目标关键词，比如“蔬菜”，它还会加过滤条件：

```text
target.name 包含 蔬菜
或 target.category 包含 蔬菜
```

所以“鸡肉配什么蔬菜好”的检索过程大致是：

```text
鸡肉
-> 使用鸡肉的菜谱
-> 这些菜谱需要的其他食材
-> 其中 category/name 和“蔬菜”相关的节点
```

这就是多跳推理的核心：答案不一定直接写着“鸡肉适合配胡萝卜”，但图里可能存在：

```text
鸡肉 -> 宫保鸡丁 -> 胡萝卜
鸡肉 -> 大盘鸡 -> 土豆
鸡肉 -> 炒鸡丁 -> 青椒
```

这些路径聚在一起，就能支持“鸡肉常和哪些蔬菜搭配”的回答。

**6. 给路径打分**

查到路径后，代码不是直接返回，而是计算路径相关性。

打分公式大致是：

```text
relevance =
    1 / path_len
  + 路径节点平均度数加成
  + 关系类型匹配加成
```

含义是：

```text
路径越短，越可能相关；
路径上的节点越重要，分数越高；
路径中包含用户关心的关系类型，额外加分。
```

比如：

```text
鸡肉 -> 宫保鸡丁 -> 胡萝卜
```

比：

```text
鸡肉 -> 某菜谱 -> 某分类 -> 某步骤 -> 胡萝卜
```

更短，通常更可信。

然后 Neo4j 查询里会：

```text
ORDER BY relevance DESC
LIMIT 20
```

先保留前 20 条图路径。

**7. 把 Neo4j 路径解析成 `GraphPath`**

每条 Neo4j 路径会被 `_parse_neo4j_path()` 转成程序内部结构：

```python
GraphPath(
    nodes=[...],
    relationships=[...],
    path_length=...,
    relevance_score=...,
    path_type="multi_hop"
)
```

这一步相当于把数据库返回的原始 path 拆成：

```text
路径上有哪些节点？
节点有什么属性？
边是什么关系？
路径长度是多少？
路径分数是多少？
```

到这里，图检索已经拿到了结构化证据。

**8. 子图查询：围绕核心实体取局部知识网络**

如果查询类型是 `SUBGRAPH` 或 `CLUSTERING`，会走 [extract_knowledge_subgraph](/d:/project/LearnAgent/all-in-rag/code/C9/rag_modules/graph_rag_retrieval.py:350)。

它适合这种问题：

```text
川菜有什么特色？
宫保鸡丁相关信息有哪些？
适合糖尿病人的低糖川菜有哪些？
```

这类问题不是只要一条路径，而是要围绕一个核心实体拿一片局部网络。

它的 Cypher 逻辑是：

```cypher
MATCH (source)-[r*1..max_depth]-(neighbor)
```

意思是：从核心实体出发，取 1 到 `max_depth` 跳以内的所有邻居节点和关系。

例如从“川菜”出发，可能拿到：

```text
川菜
-> 属于川菜的菜谱
-> 这些菜谱需要的食材
-> 这些菜谱包含的步骤
-> 相关分类、标签、时间属性
```

然后代码会计算一些图指标：

```text
node_count: 子图节点数
relationship_count: 子图关系数
density: 子图密度
```

最后包装成：

```python
KnowledgeSubgraph(
    central_nodes=...,
    connected_nodes=...,
    relationships=...,
    graph_metrics=...,
    reasoning_chains=[]
)
```

**9. 对子图做图结构推理**

拿到子图后，会进入 [graph_structure_reasoning](/d:/project/LearnAgent/all-in-rag/code/C9/rag_modules/graph_rag_retrieval.py:408)。

设计意图是：

```text
先识别子图里有什么推理模式；
再基于模式生成推理链；
最后验证推理链是否可信。
```

代码流程是：

```python
reasoning_patterns = self._identify_reasoning_patterns(subgraph)

for pattern in reasoning_patterns:
    chain = self._build_reasoning_chain(pattern, subgraph)

validated_chains = self._validate_reasoning_chains(reasoning_chains, query)
```

不过当前实现比较简化。`_identify_reasoning_patterns()` 固定返回：

```text
因果关系
组成关系
相似关系
```

`_build_reasoning_chain()` 也只是返回模板字符串。也就是说，**这份项目展示了图结构推理的架构位置，但推理链生成还没有真正做深**。

**10. 把图路径或子图转换成 `Document`**

RAG 后续生成通常吃的是 `Document`，所以图结构还要转成文本。

如果是路径结果，走 `_paths_to_documents()`：

```text
GraphPath -> 路径描述文本 -> Document
```

路径描述大概会变成：

```text
鸡肉 --REQUIRES--> 宫保鸡丁 --REQUIRES--> 胡萝卜
```

同时 metadata 里保留：

```text
search_type = graph_path
path_length
relevance_score
path_type
node_count
relationship_count
recipe_name
```

如果是子图结果，走 `_subgraph_to_documents()`：

```text
KnowledgeSubgraph -> 子图摘要文本 -> Document
```

内容类似：

```text
关于 川菜 的知识网络，包含 N 个相关概念和 M 个关系。
```

metadata 里保留：

```text
search_type = knowledge_subgraph
node_count
relationship_count
graph_density
reasoning_chains
recipe_name
```

这一步很重要：图数据库里的结构化证据，最终被压平成 LLM 可以读的上下文文本。

**11. 最后按图相关性排序并返回 Top-K**

所有 `Document` 生成后，会进入 `_rank_by_graph_relevance()`。

当前排序逻辑很简单：

```python
sorted(documents, key=lambda x: x.metadata.get("relevance_score", 0.0), reverse=True)
```

路径查询有 `relevance_score`，所以能排序。

但子图查询生成的 Document 没有显式 `relevance_score`，只有 `graph_density` 等指标。因此子图结果在当前实现里排序能力比较弱。这也是一个可优化点。

最后：

```python
return results[:top_k]
```

返回给上层 RAG 流程。

完整串起来就是：

```text
用户问题
-> LLM 解析成 GraphQuery
-> 判断查询类型
-> 如果是多跳/路径问题：从源实体出发找路径
-> 如果是子图/聚类问题：围绕核心实体抽取局部知识网络
-> 对路径或子图计算结构信息
-> 把图结构转成 Document 文本
-> 按图相关性排序
-> 返回 Top-K 给生成模型
```

所以，图 RAG 的“检索”本质上不是“找最相似文本”，而是：

```text
找实体
找关系
找路径
找局部网络
把结构化证据组织成可生成的上下文
```

这就是它比传统 RAG 更适合复杂关系问题的原因。传统 RAG 擅长“哪段文本像这个问题”，图 RAG 擅长“哪些实体通过什么关系支持这个答案”。
