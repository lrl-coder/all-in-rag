# GraphRAG 智能烹饪助手项目面试笔记

## 1. 项目一句话

这个项目是一个面向中式菜谱问答的 GraphRAG 系统。它把原始 Markdown 菜谱转成 Neo4j 知识图谱，同时把图数据整理成文本块写入 Milvus 向量库；查询时由智能路由器判断问题复杂度，自动选择传统混合检索、图 RAG 检索或组合检索，最后由大模型基于检索结果生成答案。

面试版表达：

> 我做的是一个菜谱领域的 GraphRAG 项目。传统 RAG 擅长语义相似度检索，但对“食材、菜谱、步骤、分类、难度之间的关系”理解不够强。所以我把菜谱数据建成 Neo4j 知识图谱，并保留 Milvus 向量检索能力。系统会先分析用户问题，如果是简单做法类问题，就走混合检索；如果涉及搭配、约束、关系、多跳推理，就走图 RAG 或组合检索。

## 2. 为什么要做 GraphRAG

传统 RAG 的输入通常是一批文档 chunk，向量检索能解决“语义相似”的问题，比如：

- “番茄炒蛋怎么做？”
- “红烧肉的步骤是什么？”

但它对结构关系不敏感，遇到下面这类问题会弱一些：

- “鸡肉可以和哪些蔬菜搭配？”
- “适合新手、步骤少、制作时间短的菜有哪些？”
- “哪些菜和番茄炒蛋相似？”
- “如果我只有土豆和胡萝卜，可以做什么？”

这些问题的核心不是单纯找文本，而是要理解实体和关系：

- 菜谱需要哪些食材
- 菜谱包含哪些步骤
- 菜谱属于什么分类
- 食材和菜谱之间是否共现
- 步骤之间有没有先后顺序
- 不同菜谱之间有没有相似关系

因此项目引入 Neo4j，把菜谱知识显式建模成图结构，再结合向量检索补足语义召回能力。

## 3. 系统整体架构

项目主入口是 `code/C9/main.py`，核心类是 `AdvancedGraphRAGSystem`。启动流程非常清晰：

1. 加载配置：Neo4j、Milvus、Embedding 模型、LLM 模型等。
2. 初始化 Neo4j 数据准备模块：负责从图数据库加载菜谱、食材、步骤。
3. 初始化 Milvus 向量索引模块：负责 embedding、建集合、建索引、相似度检索。
4. 初始化生成模块：负责调用大模型生成最终回答。
5. 初始化传统混合检索模块：负责图键值检索、BM25、向量检索和融合。
6. 初始化图 RAG 检索模块：负责多跳遍历、子图提取、图结构推理。
7. 初始化智能查询路由器：负责判断应该走哪条检索链路。

核心模块：

| 模块        | 文件                           | 作用                                        |
| ----------- | ------------------------------ | ------------------------------------------- |
| 主流程编排  | `code/C9/main.py`              | 初始化系统、构建知识库、交互式问答          |
| 配置        | `code/C9/config.py`            | Neo4j、Milvus、模型、chunk 参数             |
| 图数据准备  | `graph_data_preparation.py`    | 从 Neo4j 读取图数据，构建菜谱文档和 chunk   |
| Milvus 索引 | `milvus_index_construction.py` | 建向量集合、生成 embedding、执行相似度检索  |
| 混合检索    | `hybrid_retrieval.py`          | 双层图键值检索 + 向量检索 + BM25 + RRF 融合 |
| 图 RAG 检索 | `graph_rag_retrieval.py`       | 图查询理解、多跳遍历、子图提取、关系推理    |
| 智能路由    | `intelligent_query_router.py`  | LLM 分析查询复杂度并选择检索策略            |
| 生成模块    | `generation_integration.py`    | 根据检索上下文生成最终答案                  |

## 4. 数据从哪里来

项目的数据来源是第八章的 Markdown 菜谱，例如 `data/C8/cook/dishes/...` 下的菜谱文件。

数据转换流程：

1. 读取 Markdown 菜谱。
2. 使用 LLM/Agent 解析菜谱结构。
3. 抽取实体：菜谱、食材、步骤、方法、工具、难度、分类。
4. 抽取关系：菜谱需要食材、菜谱包含步骤、菜谱属于分类等。
5. 输出成两个 CSV：
   - `data/C9/cypher/nodes.csv`
   - `data/C9/cypher/relationships.csv`
6. 使用 `neo4j_import.cypher` 导入 Neo4j。

这个设计的好处是：原始 Markdown 是非结构化或半结构化文本，而 GraphRAG 需要明确的实体和关系，所以先把文档知识结构化，再进行图检索。

## 5. 图数据建模

### 5.1 节点类型

Neo4j 中主要有这些节点：

| 节点              | 含义         | 典型属性                                                                                    |
| ----------------- | ------------ | ------------------------------------------------------------------------------------------- |
| `Recipe`          | 菜谱         | `name`、`difficulty`、`category`、`cuisineType`、`prepTime`、`cookTime`、`servings`、`tags` |
| `Ingredient`      | 食材         | `name`、`category`、`amount`、`unit`、`isMain`                                              |
| `CookingStep`     | 烹饪步骤     | `description`、`stepNumber`、`methods`、`tools`、`timeEstimate`                             |
| `Category`        | 分类         | `name`                                                                                      |
| `ConceptType`     | 概念类型     | `name`                                                                                      |
| `DifficultyLevel` | 难度等级     | `name`、`level`                                                                             |
| 其他概念节点      | 方法、工具等 | `name`、`description`、`conceptType`                                                        |

`nodeId < 200000000` 的节点通常是概念层级节点，`nodeId >= 200000000` 的节点通常是具体实例节点。

### 5.2 核心关系

| 关系                                                | 含义             | 说明                          |
| --------------------------------------------------- | ---------------- | ----------------------------- |
| `Recipe -[:REQUIRES]-> Ingredient`                  | 菜谱需要食材     | 关系上保存 `amount` 和 `unit` |
| `Recipe -[:CONTAINS_STEP]-> CookingStep`            | 菜谱包含步骤     | 关系上保存 `stepOrder`        |
| `Node -[:BELONGS_TO_CATEGORY]-> Category`           | 节点属于分类     | 从 `category` 属性派生        |
| `Node -[:HAS_CONCEPT_TYPE]-> ConceptType`           | 节点属于概念类型 | 从 `conceptType` 属性派生     |
| `Recipe -[:HAS_DIFFICULTY_LEVEL]-> DifficultyLevel` | 菜谱难度         | 根据 `difficulty` 派生        |
| `CookingStep -[:NEXT_STEP]-> CookingStep`           | 步骤顺序         | 根据 `stepNumber` 派生        |
| `Node -[:SIMILAR]-> Node`                           | 相似节点         | 同类型、同分类节点之间派生    |
| `CookingStep -[:USES_SAME_TOOL]-> CookingStep`      | 使用相同工具     | 根据 `tools` 派生             |
| `CookingStep -[:USES_SAME_METHOD]-> CookingStep`    | 使用相同方法     | 根据 `methods` 派生           |

面试时可以强调：这个图不是简单导入节点和边，还根据属性额外派生了分类、相似、步骤顺序、同工具、同方法等关系，这些关系是后续多跳推理和推荐类查询的基础。

## 6. Neo4j 导入脚本做了什么

对应文件：`data/C9/cypher/neo4j_import.cypher`。

脚本按顺序做了这些事：

1. 删除旧约束，创建核心唯一约束：
   - `Recipe.nodeId`
   - `Ingredient.nodeId`
   - `CookingStep.nodeId`

2. 创建普通索引和全文索引：
   - 普通索引用于加速 `name`、`difficulty`、`cuisineType` 等字段查询。
   - 全文索引用于关键词检索，例如 `recipe_fulltext_index` 覆盖菜名、描述、标签等字段。

3. 从 `nodes.csv` 创建节点：
   - 根据 `labels` 字段创建 `Recipe`、`Ingredient`、`CookingStep` 等节点。
   - 对具体实例只处理 `nodeId >= 200000000`。

4. 从 `relationships.csv` 创建关系：
   - `801000001` 映射为 `REQUIRES`
   - `801000003` 映射为 `CONTAINS_STEP`
   - 其他关系根据规则映射或保留原始编码。

5. 基于属性派生关系：
   - `category` 生成 `Category` 节点和 `BELONGS_TO_CATEGORY`。
   - `conceptType` 生成 `ConceptType` 节点和 `HAS_CONCEPT_TYPE`。
   - 同分类生成 `SIMILAR`。
   - 连续步骤生成 `NEXT_STEP`。
   - 同工具生成 `USES_SAME_TOOL`。
   - 同烹饪方法生成 `USES_SAME_METHOD`。

6. 生成统计和计算属性：
   - 节点统计
   - 关系统计
   - `ingredientCount`
   - `stepCount`
   - `totalTime`

面试版表达：

> 这个 Cypher 脚本不是只做 CSV 导入，而是完成了知识图谱初始化。它先保证节点唯一性和查询效率，再导入菜谱、食材、步骤等核心实体，接着创建“需要食材”“包含步骤”等核心关系，最后根据分类、工具、方法、步骤序号派生增强关系，让图谱可以支持多跳推理和关联发现。

## 7. 图数据如何进入向量库

这里是项目的关键逻辑：Milvus 里并不是直接存整张图，也不是存 Neo4j 的原始边，而是存“从图数据库构建出来的菜谱文档 chunk”。

流程在 `graph_data_preparation.py` 中：

1. 从 Neo4j 加载 `Recipe` 节点。
2. 对每个菜谱查询它的食材：
   ```cypher
   MATCH (r:Recipe {nodeId: $recipe_id})-[req:REQUIRES]->(i:Ingredient)
   ```
3. 查询它的步骤：
   ```cypher
   MATCH (r:Recipe {nodeId: $recipe_id})-[c:CONTAINS_STEP]->(s:CookingStep)
   ```
4. 拼成结构化文档：
   - 标题：菜谱名
   - 菜品描述
   - 菜系、难度、时间、份量
   - 所需食材
   - 制作步骤
   - 标签
5. 按章节或长度切成 chunk。
6. 每个 chunk 保留元数据：
   - `node_id`
   - `recipe_name`
   - `category`
   - `cuisine_type`
   - `difficulty`
   - `chunk_id`
   - `parent_id`

这样做的好处是：向量库负责语义召回，元数据又能关联回图节点，实现“文本语义 + 图结构”的双重上下文。

## 8. Milvus 向量索引设计

对应文件：`milvus_index_construction.py`。

### 8.1 向量化模型

项目使用：

```text
BAAI/bge-small-zh-v1.5
```

维度是 512，适合中文语义检索。

### 8.2 Milvus Schema

核心字段：

| 字段           | 作用                  |
| -------------- | --------------------- |
| `id`           | 主键，一般是 chunk_id |
| `vector`       | 文本 embedding        |
| `text`         | chunk 原文            |
| `node_id`      | 对应 Neo4j 菜谱节点   |
| `recipe_name`  | 菜谱名                |
| `node_type`    | 节点类型              |
| `category`     | 分类                  |
| `cuisine_type` | 菜系                  |
| `difficulty`   | 难度                  |
| `doc_type`     | 文档类型              |
| `chunk_id`     | chunk 编号            |
| `parent_id`    | 父节点 ID             |

### 8.3 相似度检索基于什么

相似度检索基于 `vector` 字段。

构建索引时：

```python
texts = [chunk.page_content for chunk in chunks]
vectors = self.embeddings.embed_documents(texts)
```

查询时：

```python
query_vector = self.embeddings.embed_query(query)
```

然后用 Milvus 在 `vector` 字段上搜索。

### 8.4 索引策略

当前代码中使用：

- 索引类型：`HNSW`
- 相似度：`COSINE`
- 参数：`M=16`，`efConstruction=200`
- 查询参数：`ef=64`

亮点：

- Milvus 比 FAISS 更适合服务化和持久化场景。
- Schema 中保留了图节点元数据，支持后续过滤和回溯。
- 使用 HNSW 支持高效近似最近邻检索。
- `COSINE` 适合归一化 embedding 的语义相似度比较。

## 9. 检索策略设计

项目不是单一路径检索，而是三类策略并存。

### 9.1 传统混合检索

对应文件：`hybrid_retrieval.py`。

当前实现是三路召回：

1. 图键值双层检索
   - 实体级：菜名、食材、工具等具体实体。
   - 主题级：减肥、川菜、低脂、快手菜等抽象主题。

2. 向量检索
   - 从 Milvus 找语义相似 chunk。
   - 再根据 `node_id` 回 Neo4j 取邻居信息，增强上下文。

3. BM25 关键词检索
   - 使用 jieba 分词。
   - 使用中文停用词过滤。
   - 解决向量检索可能漏掉精确关键词的问题。

最后用 RRF 融合：

```text
score(d) = Σ 1 / (k + rank_i(d))
```

亮点：

- 向量检索擅长语义相似。
- BM25 擅长精确词匹配。
- 图键值检索擅长结构化实体和主题匹配。
- RRF 不依赖不同检索器的分数尺度，直接用排名融合，更稳。

面试版表达：

> 传统混合检索不是简单向量召回，而是图键值双层检索、Milvus 向量检索和 BM25 三路召回，再用 RRF 融合。这样可以同时覆盖语义匹配、关键词匹配和图结构匹配，降低单一路径召回失败的风险。

### 9.2 图 RAG 检索

对应文件：`graph_rag_retrieval.py`。

图 RAG 适合复杂关系问题，流程是：

1. LLM 理解查询意图。
2. 判断图查询类型：
   - `entity_relation`：实体关系查询
   - `multi_hop`：多跳推理
   - `subgraph`：子图提取
   - `path_finding`：路径查找
3. 根据类型执行图遍历。
4. 将路径或子图转成 Document。
5. 按图相关性排序。

典型例子：

用户问：“鸡肉配什么蔬菜好？”

图 RAG 可以走：

```text
鸡肉 -> 被哪些菜谱需要 -> 这些菜谱还需要哪些蔬菜 -> 蔬菜分类
```

这就是传统向量检索不容易稳定完成的多跳关联发现。

亮点：

- 能沿 Neo4j 图结构做多跳推理。
- 能抽取与问题相关的知识子图。
- 结果更可解释，因为可以给出路径或相关实体。

### 9.3 组合检索

组合检索需要两种策略结合，代价最高。问题的意图不够明确，既可能需要文本相似度召回，也可能需要图关系召回，因此用组合检索提高鲁棒性。但这是一种效果优先策略，不是成本最优策略。

流程：

1. 一部分结果来自传统混合检索。
2. 一部分结果来自图 RAG 检索。
3. 两路结果交替合并，并去重。

这样既保留语义召回的广度，又保留图推理的深度。

## 10. 智能查询路由

对应文件：`intelligent_query_router.py`。

路由器的核心作用是：不同问题走不同检索策略。

它会让 LLM 从四个维度分析用户问题：

| 维度                     | 含义         |
| ------------------------ | ------------ |
| `query_complexity`       | 查询复杂度   |
| `relationship_intensity` | 关系密集度   |
| `reasoning_required`     | 是否需要推理 |
| `entity_count`           | 实体数量     |

然后输出：

```json
{
  "recommended_strategy": "hybrid_traditional | graph_rag | combined",
  "confidence": 0.85,
  "reasoning": "推荐理由"
}
```

决策逻辑：

- 简单直接的信息查找：走 `hybrid_traditional`
- 复杂关系推理和知识发现：走 `graph_rag`
- 需要两种策略结合：走 `combined`

如果 LLM 解析失败，会降级到规则分析：

- 包含“为什么、关系、影响、比较、区别”等词，复杂度更高。
- 包含“配、搭配、组合、相关、联系”等词，关系密集度更高。

亮点：

- 不是所有问题都强行走 GraphRAG，避免复杂链路带来的成本和延迟。
- 简单问题用混合检索更快。
- 复杂问题用图结构推理更准。
- 有降级机制，LLM 路由失败时不会让系统直接不可用。



## 11. 回答生成

对应文件：`generation_integration.py`。

生成模块的输入是：

- 用户问题
- 检索到的 Document 列表

它会把文档内容拼成上下文，让 LLM 根据上下文回答。

特点：

- 支持普通生成。
- 支持流式生成。
- 流式失败时尝试切换到非流式后备。
- 当前项目已切换为 DeepSeek 兼容 OpenAI SDK 调用。

生成阶段不是项目最核心的创新点，面试时重点应该放在“图数据建模 + 多路检索 + 智能路由”。

## 12. 一条完整查询链路

以“怎么做番茄炒蛋？”为例：

1. 用户输入问题。
2. 智能路由器判断这是简单菜谱步骤查询。
3. 推荐策略为 `hybrid_traditional`。
4. 混合检索执行：
   - 图键值检索找“番茄”“鸡蛋”“番茄炒蛋”等实体。
   - Milvus 向量检索找语义相近 chunk。
   - BM25 精确匹配关键词。
   - RRF 融合结果。
5. 生成模块基于召回的食材、步骤、技巧生成回答。

以“鸡肉可以搭配哪些蔬菜？”为例：

1. 路由器识别为关系型查询。
2. 可能推荐 `graph_rag` 或 `combined`。
3. 图 RAG 从“鸡肉”出发：
   - 找需要鸡肉的菜谱。
   - 找这些菜谱还需要的其他食材。
   - 过滤出蔬菜类食材。
4. 生成模块总结搭配建议。

## 13. 项目亮点总结

### 亮点 1：从传统 RAG 升级到 GraphRAG

不是只做文档向量检索，而是显式构建菜谱、食材、步骤、分类、难度之间的图关系。这样系统可以回答关系型和多跳推理问题。

### 亮点 2：图数据和向量数据打通

Neo4j 存结构关系，Milvus 存文本语义向量。每个向量 chunk 通过 `node_id`、`parent_id` 回连到图节点，既能做语义召回，又能做图扩展。

### 亮点 3：多路召回与 RRF 融合

当前混合检索不是单一向量检索，而是：

```text
图键值双层检索 + Milvus 向量检索 + BM25 关键词检索 -> RRF 融合
```

这比纯向量检索更稳，能同时覆盖语义、关键词和结构化关系。

### 亮点 4：智能查询路由

系统会根据问题复杂度动态选择检索策略：

- 简单问题走混合检索。
- 复杂关系问题走图 RAG。
- 中等复杂问题走组合检索。

这体现了工程上的成本意识：不是所有问题都用最重的链路。

### 亮点 5：面向图推理的派生关系

Cypher 导入脚本会派生 `SIMILAR`、`NEXT_STEP`、`USES_SAME_TOOL`、`USES_SAME_METHOD` 等关系。这些关系不是原始文本直接给出的，而是为了图推理补充的结构化能力。

### 亮点 6：可解释性更强

图 RAG 可以给出实体、关系和路径，相比纯向量检索的“相似文本命中”，更容易解释为什么推荐某个菜谱或食材搭配。

## 14. 可以主动讲的工程取舍

### 为什么不用纯向量检索？

纯向量检索擅长找相似文本，但不擅长稳定回答关系问题。例如“鸡肉搭配什么蔬菜”需要从鸡肉跳到菜谱，再从菜谱跳到其他食材，还要识别蔬菜分类。图结构更适合这类问题。

### 为什么还保留 Milvus？

图检索适合结构关系，但用户自然语言问题有很多表达方式。Milvus 能做语义召回，解决“用户说法”和“数据表述”不完全一致的问题。

### 为什么加 BM25？

向量检索可能会漏掉精确词，BM25 对菜名、食材名这类关键词很有价值。比如用户明确问“番茄炒蛋”，关键词匹配可以给强信号。

### 为什么用 RRF？

不同检索器的分数不可直接比较。向量分数、BM25 分数、图匹配分数不是同一尺度。RRF 基于排名融合，不需要调权重，工程上更稳。

### 为什么需要智能路由？

GraphRAG 链路更复杂、成本更高。如果简单问题也走图推理，会增加延迟和不必要的复杂性。路由器可以让系统按问题选择合适策略。

## 15. 面试常见追问

### Q1：你的图谱是怎么构建的？

答：

> 原始数据是 Markdown 菜谱。我先通过 LLM Agent 抽取实体和关系，生成 `nodes.csv` 和 `relationships.csv`。然后用 Cypher 脚本导入 Neo4j，创建菜谱、食材、步骤、分类、难度等节点，以及 `REQUIRES`、`CONTAINS_STEP`、`BELONGS_TO_CATEGORY` 等关系。导入后还会派生相似关系、步骤顺序关系、同工具和同方法关系，增强图推理能力。

### Q2：向量库里存的是图数据库的哪些内容？

答：

> 向量库不是直接存整张图，而是存从图中构建出来的菜谱文档 chunk。每个 chunk 包含菜谱基本信息、食材列表、制作步骤、标签等文本内容。向量是基于 `chunk.page_content` 生成的，同时保存 `node_id`、`recipe_name`、`category`、`difficulty` 等元数据，用于回连 Neo4j 和过滤。

### Q3：相似度检索基于什么字段？

答：

> 基于 Milvus 的 `vector` 字段。构建时用 BGE 模型把 `chunk.page_content` 向量化，查询时把用户问题也向量化，然后用 COSINE 相似度在 `vector` 字段上检索。`recipe_name`、`category`、`difficulty` 这些字段不是相似度计算字段，主要用于元数据过滤和结果展示。

### Q4：GraphRAG 相比传统 RAG 提升在哪里？

答：

> 传统 RAG 主要靠文本相似度，能回答“怎么做”这类简单问题。但 GraphRAG 显式建模了菜谱、食材、步骤、分类之间的关系，可以做多跳推理。例如从“鸡肉”找到相关菜谱，再找到这些菜谱共现的蔬菜，最后总结搭配建议。这类关系型问题是图结构的优势。

### Q5：智能路由怎么做？

答：

> 路由器会让 LLM 分析问题的复杂度、关系密集度、是否需要推理、实体数量，然后选择 `hybrid_traditional`、`graph_rag` 或 `combined`。如果 LLM 输出解析失败，会降级到关键词规则，保证系统可用。

### Q6：项目中最有技术含量的点是什么？

答：

> 我认为有三个点：第一是把 Markdown 菜谱结构化成 Neo4j 知识图谱；第二是把图数据构造成可向量化的文档 chunk，并通过 `node_id` 保持图和向量库的关联；第三是查询阶段使用智能路由和多路召回融合，不同问题走不同策略，而不是一条链路打天下。

## 16. 项目不足和可优化方向

面试中如果问不足，可以主动说：

1. LLM 输出 JSON 不稳定
   - 路由和图查询理解依赖 LLM 返回 JSON。
   - 实际可能返回 Markdown 代码块，需要做 JSON 提取和容错。

2. 图谱质量依赖抽取质量
   - Markdown 到图数据是通过 LLM Agent 转换的。
   - 如果实体抽取或关系抽取有误，后续图推理会受影响。

3. 关系类型还可以更丰富
   - 当前核心关系是菜谱-食材、菜谱-步骤、分类、难度等。
   - 可以继续加入替代食材、口味、营养、禁忌、适用人群等关系。

4. 图 RAG 排序可以继续优化
   - 当前可以基于路径长度、节点类型、关系权重、语义相似度做更细粒度排序。

5. 评测体系可以补强
   - 可以构建一组简单查询、关系查询、多跳查询，对比传统 RAG 和 GraphRAG 的命中率、答案准确率、可解释性。

## 17. 30 秒项目介绍

> 这是一个中式菜谱领域的 GraphRAG 项目。原始数据来自 Markdown 菜谱，我先用 LLM Agent 抽取出菜谱、食材、步骤等实体和关系，生成 CSV 后导入 Neo4j，形成知识图谱。同时，系统会从图数据库中重新构建结构化菜谱文档，切分成 chunk 后写入 Milvus 向量库。查询时，智能路由器会判断问题复杂度：简单问题走图键值、向量和 BM25 的混合检索；复杂关系问题走 Neo4j 多跳图检索；中等复杂问题走组合检索。最后由大模型基于检索结果生成答案。这个项目的亮点是把图结构推理和向量语义召回结合起来，能比传统 RAG 更好地处理食材搭配、相似菜谱、约束推荐等关系型问题。

## 18. 2 分钟项目介绍

> 这个项目的背景是传统 RAG 对简单菜谱问答效果还可以，比如“番茄炒蛋怎么做”，但对关系型问题比较弱，比如“鸡肉适合搭配哪些蔬菜”“有什么低难度、制作时间短的菜”。这些问题本质上依赖菜谱、食材、步骤、分类之间的结构关系。
>
> 所以我把原始 Markdown 菜谱通过 LLM Agent 转换成图数据，生成 `nodes.csv` 和 `relationships.csv`。然后用 Cypher 脚本导入 Neo4j，创建 `Recipe`、`Ingredient`、`CookingStep` 等节点，以及 `REQUIRES`、`CONTAINS_STEP`、`BELONGS_TO_CATEGORY` 等关系。导入脚本还会派生 `SIMILAR`、`NEXT_STEP`、`USES_SAME_TOOL`、`USES_SAME_METHOD` 等增强关系，提升图推理能力。
>
> 在检索侧，我没有只用图数据库，也保留了向量检索。系统会从 Neo4j 中取出菜谱节点及其食材和步骤，拼成结构化文档，切 chunk 后写入 Milvus。每个 chunk 的向量来自 `page_content`，同时保留 `node_id`、`category`、`difficulty` 等元数据，这样向量结果可以回连图谱。
>
> 查询阶段有智能路由。路由器会分析问题复杂度、关系密集度和推理需求，简单问题走传统混合检索，复杂问题走图 RAG，中等复杂问题走组合检索。传统混合检索里又包含图键值双层检索、Milvus 向量检索和 BM25 关键词检索，最后用 RRF 融合。这样系统既能处理语义相似问题，又能处理结构关系问题。

## 19. 最推荐强调的三句话

1. 这个项目不是“向量库套壳”，而是把菜谱知识显式建成了 Neo4j 图谱。
2. Milvus 存的是从图中构造出来的菜谱文档 chunk，并通过 `node_id` 和图数据库打通。
3. 查询阶段使用智能路由和多路召回融合，根据问题复杂度选择不同检索策略，这是项目最核心的工程设计。
